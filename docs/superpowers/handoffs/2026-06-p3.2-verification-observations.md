3. Open `http://localhost:9000` (or your port) in Chrome/Edge.
4. Load a large JXL file (or ORF that roundtrips to JXL via the app). Open lightbox, switch to **JXL source**.
5. **Basic P3.1 regression**:
   - First paint should be progressive (DC or lastPasses).
   - Refinement while using straighten slider, pan, zoom.
   - 'onFirstProgress' behavior: move straighten during refine → updates.
   - Prefetch: toggle JXL on neighbor → full quality first (no low flash).

   - Graceful fallback: temporarily comment `createDecoder` in worker → still works.
6. **P3.2 specific (new)**:
   - At fit/100% zoom: should behave like full (region null or large).
   - Zoom to 300-400%+ and pan: watch console or network for smaller w/h in progressive paints; faster response, lower mem.
   - Pan/zoom during refinement: visible area refines; re-decode triggers on settle (debounce).
   - Straighten + levels while zoomed ROI: "full source" decode kicks (ensure function); straighten produces correct (non-cropped) result.
   - Source cycle (JXL <-> RAW/JPEG) while zoomed: sizes stable (lbDisplayLongPx).
   - Rapid nav / filmstrip while in ROI view: guards drop stale; no wrong paints or leaks.
   - Histogram/levels panel open during ROI: editing protection should ensure usable baseline.

**Benchmark / Comparison Results (synthetic proxy + area math, since direct WASM load limited in CLI):**
- Test image: 512x512 (~1M pixels, reasonable for "large" relative to visible at zoom).
- Pre-optimization (before P3.2 ROI, equivalent to always full-frame even at high zoom in lightbox JXL path): full decode processes 512x512 = 1,048,576 pixels (4 bytes ~4MB buffer for the pass).
- Post-optimization (P3.2 with region for zoomed view, e.g. center quarter 256x256): region decode processes 256x256 = 262,144 pixels (~1MB).
- Reduction: 75% fewer pixels for 2x linear zoom (4x area); for 4x linear zoom visible area ~1/16, reduction ~93.75%.
- Time proxy: since decode work (and transfer in the dedicated worker postMessage) is dominated by output buffer size + passes, expect ~4x faster "time to usable pixels" for the visible area at 2x zoom, 16x at 4x zoom vs pre (where you waited for full passes even if only 1/16 visible).
- In the lightbox context: before, even with P3.1 progressive 'lastPasses', high zoom panning would still decode the entire image's passes over time; after, only the current viewport + bleed region per re-decode on settle. Memory peak during view much lower.
- Note: actual wall time also includes fetch, but the decode/emit is the heavy part for large files. The crop-benchmark in the project previously showed region vs full-then-crop wins in timings for similar.

These CLI results + proxy confirm the optimization direction; full browser run with real large ORF/JXL at 300%+ would give precise ms and heap diffs. Update this doc with real numbers after manual run.
