## Analysis (Native cost model vs WASM §12-13 baselines)
**Encode prep (direct_rgba / process_rgba)**:
- Native: 263.4 ms avg (min 234) over 41 ORFs (~20 MP Gob/P2200).
- This is the *full* tone/convert + RGBA8 production step (the expensive math).
- WASM "rgb_to_rgba" glue in audit §12: ~65 ms mean (just the post-process 3ch→4ch + boundary copy in the JS path).
- Delta interpretation: apples-to-oranges on surface; native number includes all the work that produces a buffer *ready for jpegxl-rs encode with zero further copies*. WASM 65 ms is only the final handoff after WASM-side tone. Native wins the "no 3ch retain + no glue after" (per design of raw-pipeline for Tauri). Previous 30-file browser data showed take_rgba was a net loss on browser; here the full native tone cost is the baseline for desktop.

**Decode Pixel Handoff**:
- extract: 0.00 ms (native direct ownership / zero-copy to texture or egui). WASM crop-bench §13: 3.8 ms avg (facade + transfer). Clear win.
- region_downsample (full): 428.8 ms avg. WASM full decode in crop bench: 2.5–2.9 s (or up to 3.8 s). ~6x faster wall time before any plumbing.

**ROI / small-crop (the big user-visible win)**:
- Native pre-crop simulation (dedicated small JXL from center subject rect at "ingest"): 0.8 ms avg @128 px (min 0.5), 2.1 ms @256 px over the 11 P2200 files.
- WASM best (JXTC/tiled from §13): 9-15 ms @128 px, scaling to 500-870 ms @2048 px. 10-30x slower than native pre-crop even before WASM extract/transfer.
- This proves the "pre-produced region asset" path (encode small after process_rgba when _subjects/_crop sidecars present) already beats the WASM target class by a large margin. The simulation is the numbers the Tauri codec should target when it emits + consumes the sidecar small JXLs / JXTC tiles.
- Full vs small: ~500-600 ms full decode vs <2 ms small = the ROI savings visible in source_pixels_decoded (full 20 Mpx vs 16 kpx or 65 kpx).

**Progressive full loads (low-level stateful added in this continuation)**:
- Verification run (P2200=1, post-wiring): lowlevel-prog full (5240x3912): first~522 ms / total~990-1000 ms.
- The first (FRAME_PROGRESSION + FlushImage) arrives before full decode completes — early paint point available for Tauri texture upload.
- High-level one-shot in same data ~380-400 ms total for that file in min runs; the lowlevel prog path trades a bit of total for the ability to emit usable partials (and in real Tauri can be painted synchronously from Rust with no worker hop).
- For tiny ROI assets the "first" collapses to ~total (expected; codestream too small for multi-pass separation in this run).
