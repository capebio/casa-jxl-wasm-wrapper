# Overview and features of the CasaWASM JXL wrapper

## Overview
The CasaWASM JXL wrapper is a production-grade, high-performance JPEG XL (JXL) codec suite designed for scientific and consumer imaging. It provides a unified TypeScript API that bridges the gap between high-performance native execution and WebAssembly. Developed for the Casabio biodiversity platform, it prioritizes **scientific fidelity** (preserving 16-bit depth, ICC profiles, and EXIF/XMP metadata) and **progressive performance** (showing pixels immediately while a large raw image continues to decode).

It uniquely employs an advanced **preemptive scheduler** that manages a worker pool to ensure that high-priority "visible" images always take precedence over background prefetching, significantly reducing perceived latency on image-heavy gallery surfaces.

## Feature Set

### 1. High-Performance Build Architecture
*   **Multi-Tiered WASM Matrix:** Automatically selects the fastest build for the user's environment:
    *   **Relaxed SIMD + MT:** Leverages modern Chrome/Firefox features for maximum throughput.
    *   **SIMD + MT:** Primary path for Safari and stable threaded environments.
    *   **SIMD & Scalar:** Fallbacks for restricted or non-cross-origin-isolated contexts.
*   **PGO (Profile-Guided Optimization):** Optimized WASM artifacts based on real-world training data from the Casabio image corpus.
*   **Zero-Copy WASM Writes:** Both the encoder and the fallback one-shot decoder write chunk data directly into the WASM heap (`HEAPU8.set` into a pre-allocated region) rather than creating an intermediate JS `Uint8Array` via `concatBytes`, eliminating one full-image allocation and copy per operation.
*   **Grow-only WASM Allocator:** Pixel and flushed-frame buffers in the C++ bridge are sized at `JXL_DEC_BASIC_INFO` (dimensions known) and grown with `realloc` rather than `free`+`malloc`. Subsequent `NEED_IMAGE_OUT_BUFFER` events rarely need to resize, and in-flight decoded data is never needlessly copied. (`packages/jxl-wasm/src/bridge.cpp`)
*   **Immediate Chunk Slot Release:** Both the progressive decode input queue and the encoder pixel-chunk array null each slot immediately after the chunk is copied to the WASM heap, making the `Uint8Array` backing store GC-eligible during long operations without waiting for the queue to drain or `dispose()` to be called. (`packages/jxl-wasm/src/facade.ts`)
*   **Streaming Input Encoder:** Pixel data is pushed into the WASM encoder one chunk at a time via `_jxl_wasm_enc_create_image` / `_jxl_wasm_enc_finish`. Newer bridges expose `_jxl_wasm_enc_pixels_ptr` + `_jxl_wasm_enc_advance_written`, letting JS write each chunk directly into the pre-allocated WASM pixel buffer with no per-chunk temp allocation and no bridge-side memcpy. Older WASM builds fall back to `_jxl_wasm_enc_push_chunk` automatically via capability detection. Peak encoder memory stays near ~1× pixel data instead of 2×. (`packages/jxl-wasm/src/bridge.cpp`, `packages/jxl-wasm/src/facade.ts`)
*   **Safe Pixel Allocation:** `expectedPixelBytes()` validates image dimensions, checks safe-integer bounds, and rejects allocations above 1 GiB before any WASM memory is touched. (`packages/jxl-wasm/src/facade.ts`)
*   **FFI Overhead Reduction:** Bridge function references are cached once per decode session, avoiding repeated property lookups on the WASM module object at the per-chunk hot path.
*   **Module Caching:** Compiled WASM modules are cached in IndexedDB, eliminating re-compilation time on repeat visits. In Node/Bun, the loader now passes `fs.readFile()` bytes directly to `WebAssembly.compile` instead of copying them through `ArrayBuffer.slice()` first. (`packages/jxl-wasm/src/loader.ts`)
*   **Parallel WASM Feature Probes:** `getCapabilities()` previously ran the SIMD and threads `WebAssembly.instantiate` probes sequentially. They are now launched concurrently via `Promise.all`, reducing startup probe latency on slow environments from ~2× to ~1×. Dead code (`jxl1x1` placeholder array) removed. (`packages/jxl-capabilities/src/index.ts`)
*   **`compileStreaming` Unnecessary Wrapping Removed:** `compileFromResponse` previously wrapped the cloned response in `Promise.resolve()` before passing to `WebAssembly.compileStreaming`. `compileStreaming` already accepts a `Response` directly; the wrapper was a no-op. (`packages/jxl-wasm/src/loader.ts`)
*   **Native JXL Decode Fast-Path:** `getCapabilities()` probes for native browser JXL support via a `createImageBitmap` call on a minimal JXL codestream at startup. When native decoding is available (Safari 17+, or any future browser with native JXL support), the race and direct-wrapper decode paths skip WASM libjxl entirely and route through `createImageBitmap` instead — identical to how JPEG and WEBP are decoded. This drops JXL decode latency from ~120 ms to ~5 ms at 1280 px, making JXL competitive with native JPEG in the drag-race benchmark. Encode always remains WASM (no native JXL encoder exists in browsers); session-based streaming decode is also unaffected. `paintDecodedToTileCanvas` handles both the WASM frame object and `ImageBitmap` return shapes. (`web/jxl-wrapper-lab.js`, `packages/jxl-capabilities/dist/index.js`)
*   **WASM-Side RGBA Resize (`downscale_rgba`):** The canvas-based `resizeRgba` path in `jxl-wrapper-lab.js` and `jxl-benchmark.js` previously round-tripped pixel data through GPU memory (`putImageData` → `drawImage` → `getImageData`). `getImageData` stalls the GPU pipeline and reads back up to ~100 MB for a 25 MP image — 50–150 ms on integrated graphics. Replaced by a synchronous WASM box-filter (`downscale_rgba` in `src/lib.rs`), keeping pixel data in CPU/WASM linear memory throughout. Mirrors the existing `downscale_rgb` function (4-channel variant). Compiled with `+simd128`. Expected resize speedup: 2–5× for large images. (`src/lib.rs`, `web/jxl-wrapper-lab.js`, `web/jxl-benchmark.js`)
*   **Orientation 1 Fast-Path:** `process_orf`, `process_dng`, and `apply_look` skip `pipeline::apply_orientation` entirely when `orientation == 1` (the common case), moving the RGB8 buffer instead of copying it. For a 20 MP image this avoids ~60 MB of memory traffic per decode. (`src/lib.rs`)
*   **Thumbnail from Lightbox Buffer:** The 360 px thumbnail is now downscaled from the pre-computed 1800 px lightbox buffer rather than from the full-resolution RGB16 frame. For a 20 MP image this replaces a second ~120 MB full-frame scan with a ~15 MB lightbox read. (`src/lib.rs`)
*   **Unified Look-Parameter Helper (`apply_look_params`):** The 12 `is_finite` checks for exposure/contrast/highlights/shadows/whites/blacks/saturation/vibrance/temp/tint/texture/clarity were duplicated verbatim in `process_orf`, `process_dng`, and `apply_look`. Extracted to a single private helper, eliminating the source of parameter-drift bugs between the preview and export paths. (`src/lib.rs`)
*   **Pre-Allocated `rgb_to_rgba` Buffer:** `rgb_to_rgba` now writes into a `vec![0u8; n * 4]` sized up-front rather than growing via repeated `extend_from_slice`. The fixed-size buffer enables better loop vectorisation. (`src/lib.rs`)
*   **Metadata-Only Parse (`parse_orf_metadata`):** New `parse_orf_metadata(data) → OrfMetadata` function runs only the TIFF/EXIF parse stage — no decompress, demosaic, or tonemap. Returns make, model, lens, datetime, dimensions, orientation, ISO, and GPS. Enables gallery building, batch preflight, and date/GPS/lens sorting without decoding pixel data. (`src/lib.rs`)
*   **Decode Benchmark (`bench_decode_orf`):** New `bench_decode_orf(data) → DecodeBench` function runs decompress + demosaic and returns per-stage millisecond timings. Skips tonemap, downscale, and orientation, isolating raw decoder cost for tuning WASM compiler flags or algorithm changes. (`src/lib.rs`)
*   **`apply_look` accepts native `&[u16]`:** `apply_look()` now takes `rgb16_src: &[u16]` (a flat `Uint16Array` from JS) instead of packed u16 LE bytes (`Uint8Array`). The byte-by-byte unpack (`chunks_exact(2).map(u16::from_le_bytes)`) is eliminated; a single `.to_vec()` clone is still needed for in-place unsharp-mask mutation. For repeated slider edits, `LookRenderer` remains the preferred path — it owns the buffer inside WASM and requires no JS→WASM transfer after construction. (`src/lib.rs`)
*   **`LookRenderer` — WASM-Resident Slider State:** A `#[wasm_bindgen]` struct that owns the pre-tonemapped RGB16 buffer inside WASM. The JS→WASM pixel transfer happens once at construction (`take_rgb16_lb` / `take_rgb16_thumb`); every subsequent `render(wb_r, wb_b, …sliders…)` call stays entirely inside WASM, returning only the output RGB8. When `texture` and `clarity` are both zero (the common case), the internal buffer is read without cloning — no allocation. For a 1800 px lightbox this eliminates ~6 MB of JS→WASM transfer per slider edit. JS worker state is now `{renderer: LookRenderer, outW, outH, wbR, wbB}` and `release_state` / error paths call `renderer.free()` to release WASM heap memory. (`src/lib.rs`, `web/worker.js`)
*   **`process_orf_with_flags` — Selective Output Bitmask:** Variant of `process_orf` that accepts an explicit output bitmask: `1` = full-resolution RGB8, `2` = 1800 px lightbox RGB16, `4` = 360 px thumbnail RGB16. Absent outputs have empty buffers and zero dims. Batch JXL encoding passes `1` only, skipping both downscale passes (~15 MB + ~60 MB of reads avoided). Existing `process_orf` is unchanged and always passes `7`. (`src/lib.rs`)
*   **Shared ORF Decode/Output Split (`decode_orf_raw` / `process_orf_impl`):** The ORF pipeline is split into two private functions: `decode_orf_raw` (parse → validate → decompress → demosaic → NR → WB/matrix setup, returns `OrfDecoded`) and `process_orf_impl` (conditional lightbox/thumb downscale + WB override + tonemap + orientation, keyed by output flags). Both `process_orf` and `process_orf_with_flags` delegate to this pair, eliminating duplication at the entry points. (`src/lib.rs`)

### 2. Advanced Scheduling & Flow Control
*   **Prioritized Task Lanes:** Three priority tiers—`visible`, `near`, and `background`—ensure the user's current view is always processed first.
*   **Preemption with Pause/Resume:** A "visible" task can interrupt a "background" task; the background session is *suspended in-place* — the WASM decoder state remains live in the worker heap — and resumed on the same worker once the high-priority task completes. This avoids a full decode restart and preserves partial-progressive state. (`packages/jxl-core/src/protocol.ts`, `packages/jxl-scheduler/src/scheduler.ts`, `packages/jxl-worker-browser/src/decode-handler.ts`, `packages/jxl-worker-node/src/decode-handler.ts`)
*   **Pool Pre-warming:** A configurable `prewarmSize` on `SchedulerOptions` spawns idle workers eagerly at context creation. Workers start their idle timers immediately and are reaped normally if unused. Eliminates the 100–200 ms first-image cold-start latency without holding workers alive indefinitely. (`packages/jxl-scheduler/src/pool.ts`, `packages/jxl-scheduler/src/scheduler.ts`)
*   **Pool Lifecycle Hardening:** `WorkerPool` tracks in-flight spawns via a `spawning` counter so concurrent `acquire()` and `prewarm()` calls never overshoot `maxSize`. `idle` and `active` `Set`s replace per-call `filter()` scans, making worker lookup O(1). `bind()` asserts the worker is in the reserved state before promotion, catching scheduler lifecycle bugs early. `release()` and `recycle()` are idempotent and membership-safe — stale references silently no-op. `shutdown()` awaits all in-flight spawn promises before clearing the worker map, preventing escaped late-spawned workers. Spawn failures in `acquire()` return `null` rather than rejecting, matching the pool's null-capacity contract. A forward-compatible `wireWorker()` hook wires `onError`/`onExit` when the handle exposes them, replacing the previous `onMessage`-based crash detection that fired only on message arrival. Lifecycle transitions are now extracted to dedicated methods — `spawnAndReserve()`, `transitionToActive()`, `transitionToIdle()`, `handlePrewarmSuccess()` — eliminating inline duplication across `acquire()`, `bind()`, `release()`, and `prewarm()`. A centralised `cleanupAndRemove()` path (returning `Promise<void>`) serves `recycle()`, `destroyWorker()`, and `shutdown()`, so all teardown routes are consistent. `idleWorkers` and `activeWorkers` getters return `readonly` arrays; `RESERVED_SESSION_ID` is typed `as const`. A generation token (`generation`) incremented on `shutdown()` makes stale async spawn completions explicit — late workers from a previous pool lifetime are dropped rather than registered. Failed-spawn exponential backoff (100 ms → 5 s, capped at 6 doublings) prevents factory hammering when the WASM bundle is unavailable or the browser hits resource limits. A configurable `spawnTimeoutMs` (default 15 s) wraps `factory()` in a `Promise.race` so a hung factory cannot silently consume pool capacity forever. An optional `minIdle` floor keeps a warm reserve of workers alive past their idle timeout, reducing cold-start latency without holding excess workers under low load; `reapIdle()` provides a manual pressure-relief valve. `reserve()` performs a defence-in-depth stale-worker check and returns a boolean so callers can fall through to spawning rather than returning a dead handle. `idleWorkerValues()` / `activeWorkerValues()` expose zero-allocation `IterableIterator` alternatives to the array getters for hot scheduler paths. `getMetrics()` returns monotonic counters (spawned, spawnFailed, acquiredIdle, acquiredSpawned, acquireMiss, released, recycled, reaped, maxObservedSize, maxObservedSpawning) and `healthSnapshot()` returns a live per-worker state map, enabling evidence-based tuning of pool parameters. Development builds (NODE_ENV ≠ "production") activate `assertInvariants()` at the end of every lifecycle method to catch double-release, stale-reference, and overspawn bugs immediately. Worker IDs are pool-local (per-instance counter) for deterministic test output. `shutdown()` is idempotent via a shared promise — concurrent callers await the same inner operation. Named constants (`RECYCLE_SHUTDOWN_TIMEOUT_MS` = 1 s, `POOL_SHUTDOWN_TIMEOUT_MS` = 5 s) replace inline magic numbers. `hasCapacity` / `capacityRemaining` are exposed as first-class getters for scheduler call sites. An architectural comment at the module top enforces the policy/hygiene boundary: scheduling decisions (priority, preemption, dedupe) belong in `Scheduler`, not `WorkerPool`. (`packages/jxl-scheduler/src/pool.ts`)
*   **Deduplication (Fan-out):** Multiple requests for the same source URL are consolidated into a single decode session to save CPU and memory.
*   **Integrated Backpressure:** Uses WHATWG/Node stream adapters to prevent memory bloat during massive transfers.
*   **Adaptive Drain HWM:** The worker's high-water mark for incoming chunk depth is dynamically scaled by an EMA of `decoder.push()` latency. Fast workers buffer more chunks (fewer drain round-trips); slow workers signal drain earlier to avoid memory build-up. The drain message also carries the current EMA latency, giving the scheduler real-time data to tune its own `pushHwm`. (`packages/jxl-worker-browser/src/decode-handler.ts`, `packages/jxl-core/src/protocol.ts`)
*   **Input Queue Safety Cap:** `onChunk()` in both browser and Node workers enforces a 128 MiB per-session byte cap on the unprocessed input queue. Empty chunks are rejected immediately. The cap prevents unbounded memory accumulation when used without the scheduler's backpressure; in normal scheduler-managed operation the adaptive HWM keeps queued bytes well under 2 MiB and this cap never fires. (`packages/jxl-worker-browser/src/decode-handler.ts`, `packages/jxl-worker-node/src/decode-handler.ts`)
*   **`onPause` Idempotency:** Both decode handlers guard against repeated `decode_pause` messages with an early `if (this.paused) return` check, preventing duplicate `decode_paused` acknowledgements to the scheduler. (`packages/jxl-worker-browser/src/decode-handler.ts`, `packages/jxl-worker-node/src/decode-handler.ts`)
*   **Cancellation Unblocks Decoder Event Iterator:** Both handlers store the active `decoder` instance as a class field. `onCancel()` (and `failSession`/`postBudgetExceeded` in the browser handler) call `disposeActiveDecoder()` after `finishSession()`. Previously, `finishSession()` woke `feedDecoder`'s `waitForChunk`/`waitForResume` promises but could not unblock `readDecoderEvents`, which is blocked inside `for await (const event of decoder.events())`. `Promise.all` would never resolve, preventing `decoder.dispose()` from running — a WASM memory leak. A `disposingDecoder` guard prevents concurrent double-dispose. (`packages/jxl-worker-browser/src/decode-handler.ts`, `packages/jxl-worker-node/src/decode-handler.ts`)
*   **Coalesced Drain Messages with Byte Awareness:** `worker_drain` messages are now gated by both chunk count and queued bytes (`BYTE_DRAIN_HWM = 2 MiB`). Sends fire on HWM-crossing edge or after a minimum interval (`DRAIN_MIN_INTERVAL_MS = 8 ms`), whichever comes first. This prevents spam after every small chunk while keeping the scheduler responsive for slow-network single-chunk delivery (where no HWM crossing ever occurs). `MsgWorkerDrain` gains optional `queueDepth`, `queuedBytes`, and `adaptiveHwm` fields for scheduler-side tuning. (`packages/jxl-worker-browser/src/decode-handler.ts`, `packages/jxl-core/src/protocol.ts`)
*   **Budget Check After `decoder.push()`:** Both handlers now check `budgetMs` after each `decoder.push()` completes in addition to the existing check on progress events. A long-running push can exceed the budget before the decoder emits any event; without this check the session would not terminate until the next progress event. (`packages/jxl-worker-browser/src/decode-handler.ts`, `packages/jxl-worker-node/src/decode-handler.ts`)
*   **Terminal-State Wakeup (`finishSession`):** A central `finishSession(state)` helper in both handlers sets the terminal state, clears the input queue, and explicitly wakes both `waitForChunk` and `waitForResume` promises. Previously, terminal paths in `readDecoderEvents` (progressive-target header, final, budget-exceeded, error) set state without calling `wake()`, leaving `feedDecoder` parked indefinitely on a stale promise — preventing `decoder.dispose()` from running and leaking WASM memory. (`packages/jxl-worker-browser/src/decode-handler.ts`, `packages/jxl-worker-node/src/decode-handler.ts`)
*   **Budget Check Before Pixel Transfer:** In the `"progress"` event handler, `checkBudget()` runs before `postMessage(msg, [pixels])`. Previously the check ran after transfer, making the `pixels` ArrayBuffer detached; `postBudgetExceeded` would send a zero-length payload. Budget is only enforced in decoder event paths (progress/budget_exceeded events) where pixel data is available; the feed loop no longer calls `finishSession("budget_exceeded")` directly, which would have sent no protocol message and left `readDecoderEvents` blocked. (`packages/jxl-worker-browser/src/decode-handler.ts`, `packages/jxl-worker-node/src/decode-handler.ts`)
*   **`time_to_first_pixel_ms` on Final-Only Decodes:** `postFirstPixelMetric()` is now called in the `"final"` event branch as well as `"progress"`. For decodes that skip progressive frames entirely (e.g., `progressionTarget: "final"`), the final pixels are the first pixels and the metric fires exactly once. (`packages/jxl-worker-browser/src/decode-handler.ts`, `packages/jxl-worker-node/src/decode-handler.ts`)
*   **`disposeActiveDecoder()` Promise Sharing:** Disposal state is tracked as `disposePromise: Promise<void> | null` rather than a boolean `disposingDecoder`. Any path that calls `await disposeActiveDecoder()` joins the same in-flight disposal promise rather than returning immediately after a fire-and-forget caller has set a flag — ensures `run()`'s `finally` block always awaits the real decoder teardown. (`packages/jxl-worker-browser/src/decode-handler.ts`, `packages/jxl-worker-node/src/decode-handler.ts`)
*   **`onPause()` Uses `isTerminal()`:** The pause guard previously checked `cancelled || paused || state === "final" || state === "error"`, missing `state === "budget_exceeded"` and `state === "cancelled"`. Replaced with `isTerminal() || paused`. (`packages/jxl-worker-browser/src/decode-handler.ts`, `packages/jxl-worker-node/src/decode-handler.ts`)
*   **`time_to_first_pixel_ms` emitted once:** A `firstPixelMetricPosted` flag ensures the metric fires only on the first progressive frame, not on every subsequent pass. (`packages/jxl-worker-browser/src/decode-handler.ts`, `packages/jxl-worker-node/src/decode-handler.ts`)
*   **Pipelined I/O Prefetch:** `fromReadableStream` in `jxl-stream` prefetches the next network chunk immediately after each chunk arrives (before awaiting `session.push`), so network delivery of chunk N+1 overlaps with scheduler backpressure resolution on chunk N. Eliminates idle I/O time during push-wait on bandwidth-constrained connections.
*   **Stream Abort Lifecycle Hardening:** `fromReadableStream` and `toReadableStream` in `jxl-stream` properly await abort cancellation with `Promise.allSettled`, cancel the reader on push failure, check abort after prefetched reads resolve (preventing missed aborts between read and push), and clean up the abort event listener on natural stream close. `toReadableStream` wires an abort listener in `start()` so a signal that fires before `pull()` is ever called still cancels the encode session correctly. `EncodeSession.chunks()` now types as `AsyncIterable<ArrayBuffer | Uint8Array>` to match actual encoder output without a copy. (`packages/jxl-stream/src/browser.ts`)
*   **`fromResponse()` Helper:** Pipes a `fetch` `Response` body directly into a `DecodeSession` — the single most common network-to-decode pattern, with a null-body guard and full abort/backpressure support inherited from `fromReadableStream`. (`packages/jxl-stream/src/browser.ts`)
*   **Execution Budgets:** Stage-based time budgets prevent long-running decodes from hanging the worker or UI.
*   **Cold-Start Message Buffering:** Messages (`decode_chunk`, `decode_close`, `decode_cancel`, `decode_pause`, `decode_resume`, and encode equivalents) that arrive after `decode_start`/`encode_start` but before the session handler is ready (while WASM is loading) are queued per-session and flushed atomically once the handler exists. Previously these were silently dropped. Bounded at 256 messages and 128 MiB per session — the byte cap prevents a single large chunk from bypassing the message-count guard. Overflow errors include the `sessionId` in the message string for easier triage. (`packages/jxl-worker-browser/src/worker.ts`, `packages/jxl-worker-node/src/worker.ts`)
*   **WASM Load Retry:** If `loadWasmModule()` rejects, `wasmLoadPromise` is reset to `null` so subsequent sessions can retry without re-creating the worker. Previously the rejected promise was cached and all future sessions on that worker immediately failed. (`packages/jxl-worker-browser/src/worker.ts`)
*   **Idempotent Worker Shutdown:** `worker_shutdown` is deduplicated via a shared `shutdownPromise`. Duplicate or concurrent shutdown messages await the same inner cleanup rather than racing to cancel sessions twice. Shutdown awaits all in-flight session starts before cancelling their handlers. In Node, the worker sends the ack then calls `port.close()` and relies on the event loop draining naturally, with an unref'd 1 s fallback `process.exit(0)` to handle stuck callbacks — replacing an immediate `process.exit(0)` that could cut pending `postMessage` delivery. (`packages/jxl-worker-browser/src/worker.ts`, `packages/jxl-worker-node/src/worker.ts`)
*   **`release_state` Handler Cancellation:** `release_state` now calls `onCancel("release_state")` on any live decode or encode handler before removing the session entry. Previously it silently deleted the map entry, leaking the handler's internal async loop and WASM decoder/encoder resources. Also clears any buffered cold-start messages for the session, and (in the Node worker) marks the session in `cancelledPendingStarts` so a concurrently in-flight backend-init start promise does not create a zombie handler after `release_state` completes. (`packages/jxl-worker-browser/src/worker.ts`, `packages/jxl-worker-node/src/worker.ts`)
*   **Duplicate and Cross-Type Session Guard:** `decode_start` and `encode_start` check all four session maps (active decode, active encode, pending decode start, pending encode start) before registering. The Node worker additionally checks the queued-message maps, closing a narrow window where a queued-message entry could survive a failed `failPendingDecode`/`failPendingEncode` call and make a session appear absent. A duplicate session ID — including cross-type collisions where a decode and encode share the same ID — is rejected immediately with a typed error message (code `DuplicateSession`). (`packages/jxl-worker-browser/src/worker.ts`, `packages/jxl-worker-node/src/worker.ts`)
*   **Worker Uncaught Error Reporting (`worker_error`):** Both workers register global error handlers (`self.addEventListener("error"/"unhandledrejection")` in the browser; `process.on("uncaughtException"/"unhandledRejection")` in Node) and post a `worker_error` protocol message so the main thread can detect silent worker crashes that occur outside any session context. The browser `error` handler guards against `event.message` being `undefined` with a `?? "Unknown worker error"` fallback. The Node handlers route through `safePostMessage()`, which suppresses posting after shutdown when `port.close()` has already been called. (`packages/jxl-core/src/protocol.ts`, `packages/jxl-worker-browser/src/worker.ts`, `packages/jxl-worker-node/src/worker.ts`)
*   **Deterministic Node Backend Init (`backendPromise`):** The Node worker shares a single `backendPromise` singleton between the startup sequence and any concurrent first-session `initBackend()` call. Without this, startup's `selectBackend()` and an arriving session's `initBackend()` could race, probing native/WASM capability twice in parallel. On success the resolved `Backend` is cached in `backend`; on failure the promise is cleared so the next session can retry. (`packages/jxl-worker-node/src/worker.ts`)
*   **Zombie-Start Prevention (`cancelledPendingStarts`):** The Node worker tracks sessions whose pending start was invalidated (by cold-start queue overflow or `release_state`) before backend init completed. The `cancelledPendingStarts` set is checked inside the start promise immediately after `initBackend()` resolves — if the session was cancelled in the interim the promise returns without creating a handler or flushing the (already cleared) queue. Similarly, if backend init throws and the session was already cancelled by overflow, the redundant `CapabilityMissing` error is suppressed. Session-level error messages (`DuplicateSession`, `CapabilityMissing`) route through `safePostMessage()` so they are suppressed if `doShutdown` is concurrently awaiting the pending start when it resolves or rejects. (`packages/jxl-worker-node/src/worker.ts`)
*   **DecodeSession Race-Safe Push:** `push()` re-checks `terminated`/`closed` after both `acquirePromise` and `scheduler.waitForDrain()` resolve — cancellation or a worker error arriving during the drain wait cannot forward a chunk to a dead session. An early throw guard at entry catches post-close calls before the first `await`. (`packages/jxl-session/src/decode-session.ts`)
*   **Pre-Aborted Signal Handling:** If an `AbortSignal` is already aborted when a `DecodeSessionImpl` is constructed, the abort handler fires synchronously in the constructor before any event listener is added, ensuring the session settles immediately rather than waiting for an event that has already fired. (`packages/jxl-session/src/decode-session.ts`)
*   **Three-Way Terminal Helpers:** Session termination is centralised in `finish()`, `finishWithError()`, and `fail()`, all calling `cleanup()` for abort-listener removal. `finishWithError()` is specific to `decode_budget_exceeded`: the frame stream ends gracefully so consumers receive all partial frames already pushed, while `done()` rejects with `BudgetExceeded`. This is distinct from `fail()`, which fails the frame stream immediately. (`packages/jxl-session/src/decode-session.ts`)
*   **EncodeSession Race-Safe `pushPixels()`:** `pushPixels()` rechecks `terminated || finished` after both `acquirePromise` and `scheduler.waitForDrain()` resolve — cancellation or a finish call arriving during either await cannot forward a chunk to a dead or completed session. Mirrors the decode-session push guard. (`packages/jxl-session/src/encode-session.ts`)
*   **EncodeSession Abort Signal Lifecycle:** The abort handler is stored as typed fields (`abortSignal`, `abortHandler`), checked for pre-aborted state in the constructor so a signal that fires before session construction still cancels immediately, and removed via `cleanup()` on both the success path (`encode_done`) and all error/cancel paths (`terminate()`). Closes a handler-reference leak when an `AbortSignal` outlives the session. (`packages/jxl-session/src/encode-session.ts`)
*   **EncodeSession Module-Level Error Code Set:** `KNOWN_JXL_ERROR_CODES` promoted to a module-scope `ReadonlySet<string>`, matching the decode-session pattern. Eliminates a per-`encode_error` array allocation in `normalizeCode()`. (`packages/jxl-session/src/encode-session.ts`)

### 3. Scientific Correctness & Fidelity
*   **Metadata Round-trip:** Byte-exact preservation of **ICC profiles**, **EXIF**, and **XMP** metadata.
*   **High Dynamic Range (HDR):** End-to-end support for 16-bit and 32-bit float pixel formats, essential for scientific raw data.
*   **Color Management:** First-class support for sRGB, Display-P3, Adobe RGB, and Rec.2020. DNG decodes use `ForwardMatrix1/2` (preferred) or inverted `ColorMatrix1/2` from the embedded TIFF tags via `choose_camera_to_srgb_matrix`; the generic Olympus `CAM_TO_SRGB` fallback is used only when both are absent. (`raw-pipeline/src/dng.rs`)
*   **Alpha Channel Integrity:** Support for transparency without forced premultiplication, maintaining raw pixel values.

### 4. Progressive UX Features
*   **Progressive Decoding:** Emits low-resolution **DC previews** and intermediate refinement passes before the final image is complete. The progressive session loop now drains all pending passes from a single pushed transport chunk and continues draining correctly after `close()`, so refinement passes and the final frame are not delayed waiting for unrelated future input. (`packages/jxl-wasm/src/facade.ts`, `packages/jxl-wasm/src/bridge.cpp`)
*   **Region-of-Interest (ROI) Decoding:** Decodes only the requested viewport of a large raw (e.g., 100MP+), dramatically speeding up gallery views.
*   **Exact-Size Viewport Decode:** `decodeViewport({ format, targetWidth, targetHeight, fitMode })` and `decodeRegionLod({ format, region, targetLongEdge })` let callers request pixel-exact output dimensions. When region dimensions are known, the facade now chooses the best power-of-two downsample factor automatically before the final JS resize, reducing wasted decode work for viewport and ROI previews. The resize path supports `rgba8`, `rgba16`, and `rgbaf32`; it precomputes interpolation axes, skips no-op resizes, and uses typed-array fast paths on little-endian hosts. Fit modes: `contain` (letterbox, may return smaller than requested), `cover` (center-crop to exact dims), `stretch` (unconditional exact resize). Callers never need a resize fallback. (`packages/jxl-wasm/src/facade.ts`)
*   **Progressive Region Honest Flags:** `DecodeFrameEvent` carries `progressiveRegion?: boolean` and `regionFallback?: "full-frame-then-crop"` to accurately describe the decode path taken. When region + progressive decode is active, the full frame is decoded then JS-cropped; `progressiveRegion: false` and `regionFallback: "full-frame-then-crop"` are set on all emitted events so callers know the true cost. (`packages/jxl-core/src/types.ts`, `packages/jxl-wasm/src/facade.ts`)
*   **Normalized Coordinate Helpers:** `normalizedToPixelExtent(norm, imageWidth, imageHeight)` and `pixelToNormalizedExtent(region, imageWidth, imageHeight)` bridge the lightbox's normalized `[0,1]` extent model and the codec's pixel-space `Region` API. Pure math — no WASM, no async. (`packages/jxl-wasm/src/facade.ts`)
*   **Preview-First Encoding:** Biases the encoder to produce a usable "first pass" quickly, enabling "instant" upload previews on slow connections.
*   **Eliminated Double-Copy in Stream Decode:** In `streamDecodeJxlSession`, each chunk was first copied by `Uint8Array.slice()` then redundantly re-copied via `chunk.buffer.slice(byteOffset, byteOffset + byteLength)` before transfer. Since `slice()` always produces an exact-span buffer, the second copy was a no-op allocation. Changed to `chunk.buffer` directly. (`web/jxl-progressive.js`)
*   **Eliminated Redundant Pre-Copy Before Facade Push:** In the one-shot comparison decode in the progressive-paint test, `decoder2.push(jxlBytes.slice())` performed a full-image copy before the facade's own internal copy (controlled by `copyInput`, default true). Removed the caller-side `.slice()` — the facade copy is sufficient. (`web/jxl-progressive-paint.js`)
*   **`decodeJxl` Exact-Span Check in Comparison Panel:** `decodeJxl()` always called `bytes.buffer.slice(byteOffset, byteOffset + byteLength)` regardless of whether the input was already an exact-span view. Added an exact-span guard (return `.buffer` directly when `byteOffset === 0 && byteLength === buffer.byteLength`). The call site that passed `jxlEnc.bytes.slice()` to `decodeJxl` was also copying unnecessarily — the function handles exact-span extraction internally — so the pre-call `.slice()` was removed as well. (`web/jxl-compare.js`)
*   **Removed Unnecessary Copies for JPEG/WebP Encodes in Comparison Panel:** `encodeViaToBlob()` uses `putImageData` + canvas `toBlob` and reads pixel data without transferring the buffer. The comparison pipeline previously passed `new Uint8Array(rgba.buffer.slice(0))` for both JPEG and WebP paths, creating a full-image copy before each call. The JXL encode that precedes them only transfers a copy of `rgba`, not `rgba` itself, so the original buffer remains valid and can be passed directly. Removed both copies. (`web/jxl-compare.js`)
*   **Zero-Copy Server Decoder (`copyInput: false`):** The `/api/jxl-crop` server handler creates a decoder then immediately pushes `sourceBytes` (freshly read from disk). The default `copyInput: true` caused the facade to copy the full source JXL into the WASM heap before starting. Since `sourceBytes` is owned exclusively and not reused after the push, passing `copyInput: false` is safe and avoids copying potentially multi-megabyte source files. (`serve.ts`)
*   **Fixed Stack-Overflow in Tauri Upload Base64 Encode:** `btoa(String.fromCharCode(...new Uint8Array(jxl)))` spreads the full JXL byte array as function arguments; engines throw `RangeError: Maximum call stack size exceeded` for files beyond ~100 k bytes. Replaced with an explicit loop that builds the binary string incrementally, matching the standard safe pattern for arbitrary-size binary-to-base64 conversion. (`web/main.js`)
*   **Eliminated Duplicate `recommendedEffort()` Call in Legacy Encode Worker:** The one-shot encode handler computed `effortLevel = effort ?? recommendedEffort()` at the start, then called `recommendedEffort()` a second time unconditionally when building the response message (`effortRequested: effort ?? recommendedEffort()`). Changed `effortRequested` to reuse the already-computed `effortLevel`. (`web/jxl-worker.js`)
*   **Sidecar Thumbnails:** Integrated support for extracting or generating small sidecar previews during the main decode loop.
*   **Compression Ratio Feedback:** After encode completes, `encoder.getStats()` returns `{ originalBytes, compressedBytes, ratio }` (ratio = compressed / raw pixels), letting callers decide whether to keep the JXL, fall back to the original, or adjust quality before storing.
*   **Tier-Aware Effort Default:** `recommendedEffort()` maps the detected WASM tier (scalar / simd / simd-mt / relaxed-simd-mt) to a sensible encoder effort level (4 → 7), preventing scalar workers from stalling while allowing multi-threaded builds to use full quality.

### 5. Platform & Persistence
*   **Unified Cross-Platform API:** Identical interface for Browser and Node.js environments.
*   **Native Node.js Fallback:** High-speed N-API bindings for server-side processing, with a WASM fallback for restricted cloud environments.
*   **Two-Layer Caching:**
    *   **Hot In-Memory LRU:** Instant retrieval of recently viewed frames.
    *   **Persistent Cache:** Uses **OPFS (Origin Private File System)** in browsers and the local filesystem in Node for durable storage. Key features: safe OPFS filename encoding (percent-encoding via `safeCacheName`), inflight get/set deduplication (concurrent requests for the same key collapse to one OPFS read/write), oversized-item guard (items exceeding `persistentLimit` are kept in memory only), a JSON manifest (`__jxl_cache_manifest.json`) that survives page reloads making the OPFS cache warm across sessions without a directory scan, zero-length file guard on read (rejects truncated OPFS writes), aggressive quota recovery (evicts 75% on `QuotaExceededError`), and coalesced manifest writes (multiple concurrent sets batch into one write per microtask). `stats()` exposes live inflight counts and lifetime hit rate. (`packages/jxl-cache/src/browser.ts`)

### 6. Developer & Debugging Tools
*   **Comprehensive Benchmark UI:** A built-in dashboard for testing throughput across different WASM tiers and file types. The drag-race mode keeps `JXL`, `WEBP`, and `JPEG` exclusive per run, and the lab also includes an `auto-race` flow that cycles thumbnail sizes while reusing a selected folder or prompting for one if none is set. After a batch run, `drawBatchGraphs()` renders a scatter plot of per-image decode times and a bar chart of average encode/decode times (session vs. wrapper) using Chart.js. (`web/jxl-wrapper-lab.js`)
*   **Progressive Paint Test Page (`jxl-progressive-paint.html`):** Standalone test page (own nav tab) for measuring progressive decode performance. Accepts any supported source file (ORF, JPEG, PNG, TIFF, JXL) or loads a random Gobabeb ORF. Encodes progressively at user-selected size/quality/pass-count, then runs two back-to-back decodes: (1) streaming progressive — bytes fed in N equal steps via `streamIntoDecoder` with `emitEveryPass=true`, so each DC/AC pass arrives as the matching byte slice is pushed; (2) one-shot — all bytes pushed at once with `emitEveryPass=false` for a baseline final-frame time. Each progressive pass is rendered into three viewport canvases (A: overview fit, B: 1:1 pixels, C: 4× zoom) assigned automatically as passes arrive, and appended to a clickable thumbnail timeline strip. A comparison table reports stream step count, pass count, first-frame latency, final-frame latency, and the 1st-frame speedup multiplier over one-shot. (`web/jxl-progressive-paint.html`, `web/jxl-progressive-paint.js`)
*   **Crop Benchmark Page (`jxl-crop-benchmark.html`):** Standalone test page (own nav tab) for measuring thumbnail decode latency across five output sizes. User picks a local ORF folder once via `showDirectoryPicker()`; the handle is persisted in IndexedDB and restored across sessions. On each run, N random ORF files (configurable, default 3) are processed: ORF → RGBA (raw WASM pipeline), RGBA → JXL (`createEncoder`, configurable effort/distance), then five output sizes at 128 / 256 / 512 / 1024 / 2048 px are decoded sequentially via `createDecoder({ targetWidth, targetHeight, downsample })`. The page now chooses the best native 1× / 2× / 4× / 8× decode downsample from the known source dimensions before the final resize instead of always full-decoding. Each result is painted to canvas immediately as it arrives. Display columns are log₂-scaled (120 px → 400 px) so all five outputs fit side by side without large sizes dominating. Decode timings appear below each canvas (green < 80 ms, red > 400 ms) and are logged to console grouped by file. Encode time and JXL size are shown in the file row header. (`web/jxl-crop-benchmark.html`, `web/jxl-crop-benchmark.js`, `web/jxl-crop-benchmark.css`)
*   **Capability Probing:** A specialized runtime probe (`jxl-capabilities`) that detects SIMD, Threads, Relaxed SIMD support, and native JXL decode availability. Results are exposed via `getCapabilities()` as a structured capabilities object used at pipeline startup to select the optimal decode path.
*   **Debug Console Reflow Guard:** `dbgLog` previously called `listEl.scrollTop = listEl.scrollHeight` on every log entry regardless of panel visibility, forcing a browser reflow on each call. Now guarded by `!panelEl.hidden` — the reflow is skipped while the panel is closed, which is the common case during decode-heavy operations. (`web/jxl-debug-console.js`)
*   **Module Capability Detection:** `getCapabilities(module)` probes available bridge functions once per module instance and caches the result in a `WeakMap<LibjxlWasmModule, JxlCapabilities>`. Covers `progressiveDecode`, `streamingEncode`, `sidecars`, and `jpegTranscode`. Replaces per-operation `typeof` checks at call sites, ensuring fallback paths are only taken when a feature is genuinely absent. (`packages/jxl-wasm/src/facade.ts`)
*   **Telemetry Hooks:** `onMetric` callbacks provide detailed timing for "time to header," "time to first pixel," and peak memory usage.
*   **Extended Codec Metrics:** Three new `CodecMetric` discriminants surface decode-path detail: `decode_scale_used` (actual downsample factor applied), `decode_region_area` (source pixels in the requested region), `source_pixels_decoded` (total source pixels decoded). `output_bytes` is now emitted before pixel transfer in both `final` and `budget_exceeded` paths (must be read before `postMessage` detaches the buffer). `region_fallback_full_frame` (value `1`) fires once per progressive session when a region is active and the JS fallback crop path is used — previously typed but never emitted. (`packages/jxl-core/src/types.ts`, `packages/jxl-wasm/src/facade.ts`, `packages/jxl-worker-browser/src/decode-handler.ts`)
*   **`WrapperCapabilities` / `getWrapperCapabilities()`:** Static synchronous function returning a structured `WrapperCapabilities` object (`regionDecode`, `exactSizeDecode`, `progressiveRegionDecode`, `tileAlignedRegionDecode`, `arbitraryRegionDecode`, `availableDownsampleFactors`) — no WASM load required. Lightbox scheduler calls this at init to gate feature paths. (`packages/jxl-wasm/src/facade.ts`, `packages/jxl-core/src/types.ts`)
*   **`DecodeGridInfo` / `getDecodeGridInfo()`:** Placeholder returning `{}` — honest signal that libjxl exposes no tile metadata today. Lightbox checks for defined fields before using. (`packages/jxl-wasm/src/facade.ts`, `packages/jxl-core/src/types.ts`)
*   **Blur Vertical-Pass Benchmark (`blur_bench`):** `src/bin/blur_bench.rs` benchmarks candidate vertical-pass implementations for the separable Gaussian blur used by clarity/texture (13-tap kernel, 5240×3912 synthetic image). Variants: `v_pass_naive` (current production), `v_pass_tiled<TILE>` (cache-tiled, TILE = 16/32/64/128), `v_pass_clarity` (8-wide unrolled with separate scalar tail — inner `for i in 0..LANE` is always exactly 8 iterations, enabling full loop unroll unlike `v_pass_tiled` whose runtime `tile` variable may block unroll), and transpose+horizontal+transpose. Run with `cargo run --bin blur_bench --release`. (`src/bin/blur_bench.rs`)

## Comparison with Other JXL Wrappers

| Feature | **CasaWASM JXL** | **@jsquash/jxl** | **sharp** (Node) | **jxl-rs-polyfill** |
| :--- | :--- | :--- | :--- | :--- |
| **Environments** | Browser & Node.js | Browser & Edge | Node.js Only | Browser Only |
| **Progressive Support** | **Full (DC/Passes)** | Limited (One-shot) | None (Full-frame) | Limited |
| **ROI / Region Decode** | **Yes (Tile-aware)** | No | No | No |
| **WASM Optimization** | **Multi-tier + Relaxed SIMD** | Single-tier SIMD | N/A (Native) | Single-tier |
| **Scheduler** | **Preemptive / Priority / Pause-Resume** | Basic Queue | Internal (libvips) | None |
| **Metadata Fidelity** | **Scientific (EXIF/XMP/ICC)** | Partial | Good | Minimal |
| **Streaming** | Full (Backpressure) | Minimal | Native Streams | No |
| **Caching Layer** | **Integrated (OPFS/fs)** | External Only | External Only | No |

### Key Differentiators
1. **The Scheduler:** Unlike other wrappers that treat every decode as an isolated task, CasaWASM understands UI context. It can kill a background prefetch to immediately start decoding an image the user just clicked on.
2. **Region-of-Interest (ROI):** CasaWASM is optimized for multi-hundred-megapixel raws. It can decode just a "viewport" region, whereas most other WASM wrappers must decode the entire massive frame into memory first.
3. **WASM Tiers:** By shipping four distinct WASM builds (including a "Relaxed SIMD" tier), it extracts ~20-30% more performance on modern browsers than standard SIMD builds.
4. **Unified API:** Developers can write their imaging logic once and have it run with native speed on the server (N-API) and optimized WASM speed in the browser.

## Suggested and Deferred Implementations

### Bun vs. Node.js for Production
While the current architecture is built for Node.js (via N-API) and Browser (WASM), Bun represents a significant opportunity for performance optimization, particularly in the server-side ingestion pipeline.

| Metric | Bun (JSC Engine) | Node.js (V8 Engine) |
| :--- | :--- | :--- |
| **WASM Startup (Cold Start)** | **8–15 ms** | 60–120 ms |
| **Native Overhead** | **Low (via `bun:ffi`)** | Moderate (via N-API) |
| **Peak Throughput** | High | **Very High (TurboFan)** |
| **Memory Management** | **Aggressive/Efficient** | Moderate |
| **Startup Overhead** | **Minimal** | Higher |

#### Virtues of Bun Implementation
1. **Cold Start Performance:** Bun's JavaScriptCore engine initializes WASM modules up to 10x faster than Node's V8. This is ideal for CLI tools and ephemeral serverless environments.
2. **Low-Level FFI:** Migrating from N-API to `bun:ffi` could reduce the overhead of calling into `libjxl` by 2-6x, as Bun's FFI is designed for near-zero-latency C-interop.
3. **Integrated Tooling:** Bun's built-in test runner and dev server (already used in this project) drastically reduce the developer feedback loop compared to Node-based alternatives.

#### Reasons for Deferral (Current Status)
*   **Peak Throughput Stability:** For long-running, CPU-intensive JXL decodes, Node's V8 "TurboFan" compiler is currently more mature and can reach slightly higher peak throughput than Bun.
*   **Worker Thread Maturity:** Node's implementation of `node:worker_threads` remains the benchmark for stable, multi-threaded C-bridge applications at high concurrency.
*   **Ecosystem Readiness:** The project's existing `jxl-native` package is built on N-API to ensure maximum compatibility with existing Node infrastructure.

**Recommendation:** Continue using Bun for the development server, testing, and utility scripts (`bun serve.ts`, `bun test`). Transition the production server path to Bun if cold-start latency or N-API overhead becomes a primary bottleneck in high-frequency ingestion workloads.
