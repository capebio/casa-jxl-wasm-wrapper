Core Decision: Integration Shape

How the real progressive decoder (createDecoder from @casabio/jxl-wasm) enters the production lightbox without breaking straighten, live edits, filmstrip, or worker isolation.

A

Upgrade the existing dedicated worker (Handoff "Option A")

Minimal surface area. Keep the current pool + jxl-decode-worker.js as the single owner of all lightbox JXL work.

  • Extend the worker protocol with {type: 'decode_jxl_progressive', options: {progressiveDetail, region, ...}}
  • Worker does import { createDecoder } from '@casabio/jxl-wasm' (or pre-bundled) and streams events back as progress / final messages
  • Pool's decodeJxl gains optional onProgress + region params
  • Paint path in main.js stays almost identical (just handles incremental frames + calls straighten on each useful pass)
+ Preserves isolation & existing priority/dedup logic
+ Smallest diff to main.js
B

New high-priority progressive worker (Handoff "Option B")

Strong separation. Lightbox JXL gets its own worker type that loads the full progressive stack.

  • Create web/jxl-progressive-decode-worker.js that imports the real WASM + facade
  • Only used for the main lightbox canvas when mode==='jxl' (or auto-promote)
  • Thumbs + prefetch continue using the cheap jsquash one-shot worker
  • Can use jxl-session / scheduler if we want preemption & backpressure for free
+ Best long-term (real decoder gets real resources)
− New worker lifecycle + message protocol to maintain
C

Hybrid: DC-first in main + refinement in worker (pragmatic)

Fastest perceived win for P3.1. First useful pixels appear immediately; heavy refinement stays off-UI.

  • For the lightbox JXL path: fetch the bytes, createDecoder in the main thread just long enough for the first "dc" or "header" + early "progress" event, paint it, then immediately close() + transfer remaining chunks + state to a worker for the rest
  • Or simpler: do the entire progressive decode in a Web Worker but post the very first DC frame back synchronously via Atomics/SharedArrayBuffer (if COOP/COEP headers are present in Tauri/webview)
  • Still re-uses most of the existing pool callback shape
Risk: main-thread WASM time for the DC pass could still jank if the file is huge. Needs measurement.

My current lean (after exploring the code + constraints)

Start with a variant of A for P3.1 (progressive first paint). It is the smallest, safest surgical change that still delivers the "DC → refine while panning" experience. We can evolve toward B later once the event plumbing is proven in the lightbox context.

Region/ROI (P3.2) works cleanly on top of either A or B because the decoder already surfaces ev.region and the C++ bridge has fast crop.
