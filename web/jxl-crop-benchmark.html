Region decode latency by crop size

Encodes a raw file to JXL, then benchmarks createDecoder({ region }) for each crop size centred on the image.

Number of ORF source files from the picked folder to benchmark. More files give broader coverage across image content and more stable averages.
JXL encode effort (1–9). This benchmark measures decode speed; effort mainly affects file size and thus tile read I/O. Effort 3 gives a realistic photographic JXL. Does not affect visual quality.
Butteraugli perceptual quality distance. Lower = higher quality, larger file. 0 = lossless, 1.0 = high quality photographic, 3.0 = moderate quality. This is the internal quality scale (inverse of the Quality slider).
Tile size in pixels for the tiled JXL encode. Each tile is an independent frame — only tiles intersecting the crop are decoded. Smaller tiles = faster crop decode, more frame-header overhead. Try 256, 512, and 1024 to find the sweet spot.
When enabled, each crop size also runs a full-image decode for comparison. Shows how much the tiled crop saves vs. decoding the entire image, across crop sizes.
Crop sizes (px, centred):
Square crop regions centred on the image. For each size, only the tiles intersecting that crop are decoded. Smaller crops should be faster than a full decode — this benchmark measures by how much, and how it scales relative to tile size.
