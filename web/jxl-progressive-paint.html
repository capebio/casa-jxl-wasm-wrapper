Progressive Paint

Per-pass timing + visual quality

Size:
Long edge in pixels to resize the source before encoding. Smaller = faster encode+decode cycles. 1080 suits display-resolution testing; Full encodes at the original source resolution for archival comparison.
Quality:
JXL encode quality (0–100). Higher = better fidelity, larger file. 85 is visually indistinguishable from the original for most photographic content. Higher quality means more data per pass, which affects how much each progressive step improves the image.
Stream steps:
Number of byte-feed steps used during the progressive decode test. Actual paints can be lower than requested steps because libjxl only emits frames when the encoded codestream has a displayable DC/pass/final layer.
Detail:
Auto — Decoder chooses which passes to surface.
DC — Emits only the low-frequency DC pass first (fast, coarse). DC + AC is not available in libjxl v0.11.2.
Last passes — Surfaces only the final high-quality pass.
All passes — Emits every intermediate pass; maximum granularity, more canvas updates.
Inserts a small low-resolution preview frame at the very start of the encoded stream. The viewer can display this placeholder immediately, before the main progressive decode begins. Useful for measuring time-to-first-pixel.
Center-out (groupOrder=1) starts early progressive passes from image center radiating out (vs scanline strips). With progressiveDc=2 + All passes, early layers are far more recognizable for low-byte "first useful" measurement. Recommended default for benchmark passes >=4.
