## Change
Confirmed via jpeg-decoder 0.3.2 source that both `scale()` and `decoder.info()` post-decode read from the same `frame.output_size` field. The pre-fix code captured `out_w/out_h` from `scale()` but returned them without using `decoded.width/decoded.height`, which were already in scope. Changed the return to use `decoded.width as u32` / `decoded.height as u32` from the post-decode `info()` call, which is the authoritative source regardless of whether scaling was applied. Dropped the `out_w/out_h` binding entirely since the return value of `scale()` was its side-effecting config of the IDCT scaler, not needed as a value.
